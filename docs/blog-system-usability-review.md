# 博客系统易用性与现代化审查总结

本文档汇总管理员视角、普通读者视角、现代博客系统视角的只读审查结果。审查范围为当前项目代码，不包含联网调研。

## 总体结论

项目已经具备现代博客系统雏形：React/Vite 前端、FastAPI + SQLite 后端、管理台、文章编辑器、草稿、发布/下架、回收站、评论审核、图库、动态 RSS/sitemap、SEO 字段和 smoke test 均已存在。

但系统还未达到成熟生产级现代博客标准。主要短板集中在 SEO 路由一致性、SPA 收录能力、公共导航、后台发布状态反馈、安全加固、测试覆盖和前端可维护性。

## 视角评分

| 视角 | 评分 | 结论 |
| --- | --- | --- |
| 管理员易用性 | 7/10 | 后台功能覆盖较全，但保存失败反馈、安全和运维能力不足。 |
| 普通用户易用性 | 6.5/10 | 首页和阅读体验基础不错，但导航、筛选和可访问性仍需补强。 |
| 现代博客系统完整性 | 6/10 | 内容模型和核心能力已具备，但 SEO、性能、安全、测试和架构拆分不足。 |

## 主要问题

### P0：RSS/sitemap 文章路径与前端路由不一致

已修复。前端文章详情路由和服务端动态 RSS/sitemap 均使用 `/posts/:slug`。

相关位置：

- `src/App.tsx`：公开文章路由 `/posts/:slug`
- `server_py/app.py`：`/rss.xml`、`/sitemap.xml` 输出 `/posts/:slug`

建议：

- 统一所有公开文章 URL 为 `/posts/:slug`。
- 为 RSS/sitemap 路径一致性补 smoke test。
- 确认 robots、RSS、sitemap、前端链接全部使用同一套 URL。

### P0：后台保存失败可能造成“已发布/已保存”的错觉

管理员创建或更新文章时，API 失败后存在本地 fallback 和跳转行为。草稿保护是有价值的，但如果反馈不明显，管理员可能误以为内容已经同步到后端或公开发布。

建议：

- API 保存失败时不要静默跳转。
- 显示“仅本地保存，尚未同步服务器”的醒目状态。
- 提供重试同步入口。
- 在文章列表中标记未同步草稿。

### P1：文章级 SEO 未真正落地

文章模型和编辑器中已有 `seoTitle`、`seoDescription`、封面等字段，但 SPA 文章详情页没有为每篇文章生成独立 HTML，也未完整更新 `document.title`、description、canonical、Open Graph/Twitter meta。

建议：

- 短期：文章详情页运行时更新 title、description、canonical、OG/Twitter meta。
- 中期：为文章生成静态 HTML 或引入 SSR/SSG。
- 增加 JSON-LD、canonical、封面图 meta。
- 将后台 SEO 字段和最终页面 head 输出打通。

### P1：静态内容、localStorage 和 SQLite 内容来源双轨

已收敛。公开页面以 SQLite API 为单一事实来源；静态 `src/posts.ts` 只作为 seed 和迁移输入，不再作为公开页面渲染 fallback。API 不可用时显示数据库内容接口不可用提示。

建议：

- 明确生产环境以 SQLite 为单一事实来源。
- 静态文章只作为 seed 或迁移输入。
- localStorage fallback 需要明确标记为离线缓存，避免长期覆盖服务器内容。
- RSS/sitemap 应以后台数据库已发布文章为准。

### P1：公共导航对普通读者不够友好

公共导航中包含后台登录入口，且“文章/札记/归档”主要跳首页锚点，不直接进入完整列表页。页脚也把 `/admin` 展示为“站点设置”，容易打断普通读者。

建议：

- `文章` 指向 `/posts/page/1`。
- `札记` 指向 `/notes/page/1`。
- `归档` 指向 `/archive/page/1`。
- 后台登录入口移到更低干扰的位置，或仅保留隐性入口。

### P1：搜索和筛选体验偏弱

搜索能匹配标题、摘要、分类、标签和正文；列表页已有分类、标签、当前筛选、清除筛选和无结果状态。分类与标签现在可以组合筛选。

建议：

- 文章列表增加分类 tabs。
- 继续优化标签数量较多时的折叠与搜索。

### P1：后台安全不足

当前后台为单密码登录，默认管理密码存在生产风险；session 存在内存中；cookie 未配置生产 `secure`；CORS 使用 `origin: true` 且带 credentials；后台写操作未见 CSRF 防护；登录和评论缺少限流。

建议：

- 生产环境禁止默认管理密码启动。
- Cookie 在 HTTPS 环境启用 `secure`。
- CORS 改为明确白名单。
- 后台写操作增加 CSRF 防护。
- 增加登录失败限流、评论限流和反垃圾策略。
- 增加管理员操作审计。

### P1：公开图库路由/API 不完整

SEO 生成脚本和测试中涉及 `/gallery` 或 `/api/gallery`，但前端公开路由未覆盖 `/gallery`，后端也只有 `/api/admin/gallery`。

建议：

- 如果需要公开图库：实现 `/gallery` 页面和 `/api/gallery` 公开接口。
- 如果不需要公开图库：从 sitemap、SEO 生成和 smoke test 中移除相关入口。

### P2：可访问性需补强

搜索弹层已有 `dialog`、`aria-modal`、自动聚焦和 Escape 关闭，但缺少焦点陷阱和关闭后焦点恢复；部分输入移除了 outline；全局链接/按钮缺少统一 `:focus-visible`。

建议：

- 增加全局 `:focus-visible` 样式。
- 搜索弹层实现焦点陷阱。
- 关闭弹层后恢复到触发按钮。
- 移动抽屉使用更明确的导航语义。

### P2：文章阅读辅助信息不足

文章详情页已有面包屑、分类、标题、摘要、作者、日期、头图、正文、上一篇/下一篇、阅读时长、标签、目录、相关文章和评论。评论接口失败时本地临时保存，读者可能误以为评论已公开提交。

建议：

- 明确区分“审核后展示”和“本机临时保存”。

### P2：运维和内容治理能力不足

后台总览更偏内容数量和快捷入口，不是运维面板。当前缺少访问统计、发布失败队列、错误日志查看、备份恢复、数据库状态、任务状态、审计日志等信息。

建议：

- 增加 API 健康、数据库状态、最近发布、待审核评论数。
- 增加未同步草稿数、最近错误、最近备份时间。
- 发布前增加 SEO、摘要、slug、封面、分类检查清单。

### P3：前端核心文件过大

`src/App.tsx` 集中了承载公开页面、后台、编辑器、图库、评论、Markdown 渲染和路由逻辑，文件超过 6500 行。短期能工作，但长期维护和测试成本高。

建议：

- 按 public pages、admin、composer、gallery、comments、markdown renderer、routing 拆分模块。
- 后台编辑器使用 lazy load，避免普通读者加载编辑器依赖。
- 将 API hooks、数据归一化、路由工具拆出独立文件。

## 建议实施顺序

1. 修正 `/articles` 与 `/posts` 路由不一致问题，并补 RSS/sitemap 路径测试。
2. 明确 `/gallery` 是否公开；实现对应页面/API，或从 SEO/test 中移除。
3. 调整公共导航，让普通读者优先进入完整文章、札记和归档列表。
4. 修正后台保存失败反馈，区分服务器已保存和本地临时保存。
5. 落地文章级 SEO，并逐步评估 SSR/SSG 或静态文章 HTML 生成。
6. 加固后台安全：默认密码、CORS、cookie、CSRF、限流和审计。
7. 增强列表筛选、可访问性和文章阅读辅助信息。
8. 拆分 `src/App.tsx`，并对后台编辑器做 lazy load。
9. 补充部署、环境变量、数据库迁移/备份、SEO 生成策略文档。

## 当前亮点

- 已有完整管理台入口和文章编辑流程。
- 支持 Markdown、公式、表格、代码高亮、图库图片。
- 支持草稿、发布/下架、软删除、回收站。
- 支持评论审核。
- 支持站点外观、首页文案、分类、专题、图库管理。
- 后端已有健康检查、UTF-8 JSON 响应、上传文件类型和大小限制。
- 已有 RSS/sitemap/robots 生成与校验脚本。

## 结论

优先修复外部入口正确性和发布状态真实性，再推进 SEO、安全和架构拆分。当前项目已经具备可用博客基础，但距离稳定、可维护、可收录、可运营的现代博客系统，还需要一轮面向生产的系统性收敛。
